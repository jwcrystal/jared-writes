# 常用業務存儲系統對比

六大存儲系統各有所長，不存在「最好的」，只有「最適合場景的」：MySQL 管結構化資料和強一致性，Redis 管速度，MongoDB 管靈活 Schema，ES 管搜尋，Hadoop 管離線大數據，HBase 管即時大數據讀寫。選型的核心不是效能對比，是理解每個系統的資料模型和一致性模型。

> **選存儲系統就像選車：不是法拉利最好，是看你要載貨、通勤、還是越野。**

## 六大系統對比

| 系統 | 類型 | 資料模型 | 核心優勢 | 明顯限制 | 典型場景 |
|------|------|----------|----------|----------|----------|
| MySQL | RDBMS | 表格/SQL | ACID、複雜查詢 | 水平擴展困難 | 業務系統主力 |
| Redis | 記憶體 | KV/資料結構 | 極速、豐富資料型別 | 受記憶體大小限制 | 快取、Session、排行榜 |
| MongoDB | 文檔 | JSON/BSON | 動態 Schema、易擴展 | 舊版無 ACID（已改善） | 半結構化資料、快速迭代 |
| ES | 搜尋引擎 | JSON 文檔 | 全文搜尋、倒排索引 | 事務和 JOIN 弱 | 日誌分析、全文搜尋 |
| Hadoop | 分散式系統 | 檔案（HDFS） | 海量資料批次處理 | 即時處理弱 | 離線數據分析 |
| HBase | 列式儲存 | 稀疏矩陣（BigTable） | 大規模即時讀寫 | 複雜查詢弱 | 時序數據、大規模 KV |

## 學習順序建議

```
MySQL → Redis → MongoDB → Elasticsearch → Hadoop → HBase
```

這個順序的邏輯：
1. MySQL 打底：SQL 和關聯式模型是所有其他系統的對比基準
2. Redis 擴展視野：最簡單、最快的系統，展示了記憶體運算的極限
3. MongoDB 打破結構：引入文檔模型和動態 Schema
4. ES 引入搜尋：倒排索引的概念完全不同於 B+ 樹
5. Hadoop 進入大數據：HDFS + MapReduce 是分散式運算的基礎模型
6. HBase 收尾：建立在 Hadoop 上，展示列式儲存的設計哲學

## 各系統架構關鍵點

- **MySQL**：連接層 → 服務層（解析/優化/執行）→ InnoDB（B+ 樹 + redo/undo log + MVCC）
- **Redis**：單執行緒事件循環 + epoll I/O 多工；RDB 快照 / AOF 日誌持久化；主從複製
- **MongoDB**：mongos 路由 → config server（元資料）→ shard（副本集）
- **ES**：基於 Lucene 倒排索引；節點 → 分片（shard）+ 副本；近即時搜尋（refresh interval ~1s）
- **Hadoop**：HDFS（NameNode 管元資料 + DataNode 存資料塊）+ MapReduce
- **HBase**：HMaster 管理 + RegionServer 存資料 + ZooKeeper 協調 + HDFS 底層儲存

## 分析與建議

- 先 MySQL 後 Redis 是最自然的學習路徑，因為生產環境中它們經常搭檔：MySQL 做主儲存，Redis 做快取層。
- MongoDB 很適合 PoC 和快速迭代（不需要預先定義 Schema），但到了複雜查詢和報表場景就會想念 SQL。
- Hadoop 生態正在被雲原生方案取代（Spark 取代 MapReduce，物件儲存取代 HDFS）。學習 Hadoop 的價值在於理解分散式運算的基礎模型，而不是實務操作。
- ES 的倒排索引是理解搜尋引擎的關鍵。一旦理解了「從關鍵字找文件」vs「從文件找關鍵字」的差異，就理解了為什麼 ES 做全文搜尋比 MySQL LIKE 快幾個數量級。

## 總結

**沒有萬能的存儲系統。好的架構師不是挑一個最強的，是知道每個系統的邊界在哪，然後把它們拼在一起。**
