# 數據庫與緩存雙寫一致性

## 核心摘要

這是後端系統面試和實務中最常遇到的問題之一：當一筆數據同時存在 DB 和 Cache（如 Redis），更新時要先寫誰才能保證一致性？四種方案（先寫緩存、先寫 DB、先刪緩存、先刪緩存再寫 DB）在高併發場景下各有陷阱。目前業界公認的最優方案是 **Cache Aside Pattern + 延遲雙刪**。

## 一句話理解

**先更新 DB，再刪除緩存。寧可出現一次 cache miss（多查一次 DB），也不要出現 cache 裡是舊資料。**

## 背景 / 問題

標準的讀取流程是：
```
請求 → 查 Cache → 有就回傳 → 沒有就查 DB → 寫入 Cache → 回傳
```

但更新時的寫入順序會產生不一致問題。如果更新 DB 和更新 Cache 不是原子操作，在高併發下一定會出現時間窗口讓讀者拿到舊值。

## 四種方案對比

| 方案 | 流程 | 核心問題 |
|------|------|----------|
| 先寫緩存，再寫 DB | Cache → DB | DB 寫失敗會讓 Cache 有髒數據且無法回滾 |
| 先寫 DB，再寫緩存 | DB → Cache | 兩次寫入非原子，併發下 Cache 可能被舊值覆蓋 |
| 先刪緩存，再寫 DB | Delete Cache → Write DB | 刪完到寫完之間，其他請求讀到舊 DB 值並寫回 Cache |
| **先寫 DB，再刪緩存** | **Write DB → Delete Cache** | **最佳方案，但仍有極短的不一致窗口** |

## 最佳方案：Cache Aside Pattern

```
讀取：Cache → (miss) → DB → 寫入 Cache → 回傳
更新：更新 DB → 刪除 Cache
```

關鍵設計：**緩存是懶加載的**。更新時只刪除緩存，不主動寫入。下一次讀取會觸發 cache miss，從 DB 讀最新值再寫入。

### 延遲雙刪（Double Delete）

為了消滅極短的不一致窗口（更新 DB 後、刪除 Cache 前有併發讀）：

```
1. 刪除 Cache
2. 更新 DB
3. 延遲 N ms
4. 再次刪除 Cache
```

步驟 4 清理步驟 2-3 之間可能被其他請求寫入的舊值。延遲時間通常設定為讀取 DB + 寫入 Cache 的耗時。

## 我的判斷

- 對於 90% 的場景，**先寫 DB 再刪 Cache** 已經足夠。Cache Aside 簡單、可靠，不需要引入 MQ 或 CDC 等額外組件。
- 如果你的業務可以容忍短暫的不一致（例如用戶暱稱延遲 1 秒更新），不需要用延遲雙刪。
- 如果你的業務是金融級別的一致性要求，緩存就不該是唯一的數據來源——考慮用 DB 做主數據源，緩存只做唯讀加速。
- 面試中的關鍵是表達出「沒有完美方案，每個方案都有 trade-off」，以及「根據業務對一致性的容忍度選擇方案」。

## 最後記住這句

**Cache Aside = 讀的時候寫 Cache，寫的時候刪 Cache。簡單、有效、面試時說出來就過關。**
