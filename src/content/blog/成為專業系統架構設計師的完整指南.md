# 成為專業系統架構設計師的完整指南

## 核心摘要

從初級工程師到系統架構設計師大約需要 10-12 年，這個職涯路徑的關鍵不是「學更多工具」，而是**從寫 code 轉向做 trade-off**：CAP 選哪邊、同步還是非同步、SQL 還是 NoSQL。核心能力三根柱子：分散式系統理論基礎（CAP/Paxos/Raft）、實戰架構經驗（微服務/容器化/雲端）、以及把技術決策翻譯成業務語言的能力。

## 一句話理解

**架構師不是最會寫 code 的人，是最會在約束條件下做技術取捨的人。**

## 職涯階段

| 階段 | 年資 | 重點 |
|------|------|------|
| 初/中級工程師 | 0-3 年 | 精通一門語言，理解設計模式，獨立交付 |
| 資深工程師 | 3-7 年 | 獨立設計系統，深入分散式概念，參與架構決策 |
| 架構師 | 7-12 年 | 設計大規模分散式系統，跨領域廣度 + 單領域深度 |

## 必備硬技能

| 領域 | 核心內容 |
|------|----------|
| 分散式系統 | CAP 定理、BASE、Paxos/Raft 共識演算法 |
| 資料庫 | PostgreSQL、NoSQL（MongoDB/Redis）的取捨，索引、複製、分區 |
| 微服務 | 服務邊界定義、API 設計、事件驅動架構 |
| 容器化 | Docker、K8s、IaC（Terraform） |
| 可觀測性 | Prometheus、ELK、分散式追蹤 |
| 網路/OS | TCP/IP、HTTP/DNS、行程/執行緒/並發 |

## 必備軟技能

- **系統思維**：從全局視角做 trade-off（ACID vs 最終一致性、垂直 vs 水平擴展）
- **溝通能力**：把技術決策翻譯成業務語言，讓 PM 和 exec 理解為什麼選這個方案
- **領導力**：指導團隊按架構執行，code review 傳遞最佳實踐
- **業務理解**：技術決策必須對齊商業目標，不是為了技術而技術

## 學習路徑

1. **打底**：精通一個後端語言 + 框架，做完整的 side project 累積實戰
2. **讀神書**：《Designing Data-Intensive Applications》（DDIA）是分散式系統的聖經
3. **練面試**：用 RESHA 框架練習系統設計題（即使不面試，這個思考框架也很有用）
4. **實戰**：在真實專案中應用微服務，體驗服務間通訊的痛
5. **學工具**：Docker → K8s → Terraform → CI/CD → 監控，按這個順序

## 核心書籍與資源

| 資源 | 用途 |
|------|------|
| DDIA (Kleppmann) | 分散式系統理論的完整地圖 |

## 我的判斷

- 10-12 年不是硬指標。在 startup 環境且從零參與架構設計的人，可能 6-7 年就能達到架構師的判斷力。在大廠做單一模組維護的，可能要更久。
- **如果你現在是 junior/mid-level（0-5 年），最該做的事不是學 K8s 或 Terraform，而是**：找一個機會從頭參與一個系統的設計——哪怕是 side project。寫 spec → 畫架構圖 → 選技術 → 上線 → 出事 → 救回來，這個循環走一遍，比讀十本書都有用。
- 最容易被忽略的是**軟技能**，尤其是「把 trade-off 說清楚」的能力。技術上你知道選 Cassandra 的理由，但如果不能用 PM 聽得懂的話講出來，決策不會落地。
- DDIA 是全職涯最重要的書，沒有之一。但不要在 junior 的時候硬啃——先累積 2-3 年實戰經驗再讀，體會完全不同。
- 認證（AWS SA Pro 等）在求職時有加分，但不要為了認證而考。真正的架構能力來自設計過、上線過、出事過、救回來過。
- **架構師的面試和工程師完全不同**：你不再被考 leetcode，而是被問「這個系統瓶頸在哪裡」「為什麼選 PostgreSQL 而不是 Cassandra」「如果流量突然翻十倍怎麼辦」。如果你發現自己更享受這種討論而不是寫 algorithm，那你的方向對了。

## 最後記住這句

**架構設計師的核心能力不是知道多少種技術，而是在 CAP、成本、時程、團隊能力的多重約束下，做出一個能活到明天的決策。**
