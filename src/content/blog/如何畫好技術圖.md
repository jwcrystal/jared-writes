# 如何畫好技術圖

## 核心摘要

一張好的技術圖需要兩層功夫：**表面功夫（視覺設計）和內功心法（架構思維）**。表面功夫決定圖能不能被看懂——佈局對齊、資訊分層、顏色引導、線條整齊。內功心法決定圖值不值得被看——是否抓住了系統的核心抽象、組件邊界是否清楚、資訊層級是否合適。兩者缺一不可：視覺再漂亮但邏輯混亂的圖是誤導，架構再清楚但畫得亂七八糟的圖沒人想讀。

## 一句話理解

**技術圖不是美術作品——它是一個架構決策的可視化合約，讀者應該 30 秒內理解核心結構。**

## 表面功夫：視覺法則

| 面向 | 法則 |
|------|------|
| **佈局** | 有重心、對齊、留白約 30%；避免顆粒度混亂 |
| **分層** | 相關元素群組化，形成清晰層次 |
| **視覺引導** | 順應自然閱讀順序（左→右、上→下） |
| **色彩** | 主色 + 輔助色，同色代表同功能；文字與背景高對比 |
| **線條** | 多用直角折線；保持粗細、符號一致 |

## 內功心法：架構思維

| 面向 | 法則 |
|------|------|
| **全局理解** | 畫圖前先掌握整體系統和業務需求 |
| **抽象層次** | 找出合適的抽象層級——不過載、不過簡 |
| **模組通信** | 模組之間要有清楚的連線關係，不是孤立拼湊 |
| **邊界明確** | 每張圖定義展示範圍，不同細節分不同圖 |

## 常見陷阱

- **堆砌**：不相關或不同層級的內容塞在同一張圖，邏輯混亂
- **抽象不當**：要嘛太細節（變成一張 UML class diagram 級別），要嘛太空泛（只剩下幾個方塊和箭頭）
- **缺乏圖例**：顏色和線型的意義不解釋，讀者要靠猜

## 推薦工具

| 工具 | 適合場景 |
|------|----------|
| Excalidraw | 手繪風格、快速草圖、協作 |
| draw.io | 正式架構圖、流程圖 |
| Mermaid | 文字生成圖、版本控制友善 |

**核心不是工具本身，而是你對系統的理解。**

## 我的判斷

- 「技術圖是個人能力與系統理解力的名片」這句話說得很準。在晉升匯報、系統設計面試、跨團隊溝通中，一張圖的品質直接影響別人對你專業能力的判斷。
- 最常見的問題是**一張圖塞太多東西**。解決方案是：不同層級的問題用不同圖。不要想用一張系統全景圖解釋所有細節。
- 顏色的使用是初學者最常犯的錯：彩色不是為了好看，是為了**分組和引導注意力**。如果你用了 5 種顏色但沒有分組意義，不如只用 2 種。
- 這篇文章和 [[如何在2025年編寫出色技術文檔的指南]] 是姊妹篇——一個講文字，一個講圖。好的技術溝通是兩者兼備。

## 最後記住這句

**一張技術圖 30 秒內看不懂，不是讀者的問題，是你的抽象層級選錯了。拆成多張圖、簡化層級、統一色彩——讓架構自己說話。**
