# 消息隊列技術比較 — 主流系統對比與選型

消息隊列解決的是現代分散式系統最根本的問題：生產者和消費者之間的速度不匹配。生產者可以瞬間吐出大量消息，消費者處理不完 — 沒有隊列作為緩衝，系統就崩了。

> **消息隊列不是讓你更快，是讓你不會因為快慢不一而崩潰 — 它是生產者和消費者之間的減速帶。**

## 基本概念

```
Producer → [Broker → Queue/Topic] → Consumer
```

| 角色 | 職責 |
|------|------|
| Producer | 產生並發送消息 |
| Broker | 管理存儲、路由、副本、高可用 |
| Consumer | 取出消息並處理 |
| Queue（點對點） | 一條消息只被一個 consumer 處理 |
| Topic（Pub/Sub） | 一條消息被多個 consumer 各自處理 |

## 五個核心能力指標

| 指標 | 含義 | 影響選型 |
|------|------|----------|
| 吞吐量 | 每秒消息數 | Kafka 大幅領先 |
| 延遲 | 生產到消費的時間差 | 交易場景要求 < 10ms |
| 可靠性 | 消息不丟失的保證 | 金融支付不能接受任何丟失 |
| 擴展性 | 集群水平的線性增長能力 | Kafka 的分區設計天然優勢 |
| 順序性 | 是否保證分區內順序 | 交易必須有序 |

## 五大系統對比

| 系統 | 核心架構 | 吞吐量 | 特色優勢 | 明顯限制 |
|------|----------|--------|----------|----------|
| **Kafka** | 分區 + 副本 + 順序寫磁盤 | 最高 | 日誌/串流場景王者，生態豐富 | 依賴 ZooKeeper（正在改），分區數過多影響性能 |
| **RabbitMQ** | Exchange + Queue + AMQP | 中 | 靈活路由、多協議支援、社區成熟 | 吞吐量天花板明顯 |
| **RocketMQ** | CommitLog + ConsumeQueue 分離 | 高 | 事務消息、延時消息、順序保證強 | 社群不如 Kafka 活躍 |
| **Pulsar** | 計算存儲分離（BookKeeper） | 高 | 雲原生、多租戶、跨地域複製、彈性擴容 | 架構複雜，依賴 ZooKeeper + BookKeeper |
| **ActiveMQ** | JMS 標準 | 低 | 企業整合、跨平台 | 性能和擴展性落後一代 |

## 選型指南

| 場景 | 推薦 | 理由 |
|------|------|------|
| 大數據串流、日誌收集、事件溯源 | Kafka | 吞吐量 + 持久化 + 生態（Kafka Connect/Streams） |
| 微服務間非同步通訊、複雜路由 | RabbitMQ | AMQP 路由靈活，管理介面成熟 |
| 電商交易、金融支付（事務消息） | RocketMQ | 事務消息 + 順序保證 + 延時消息 |
| 多租戶 SaaS、跨 DC 複製、雲原生 | Pulsar | 存算分離可獨立擴展、BookKeeper 持久化 |
| 遺留系統整合 | ActiveMQ | JMS 標準兼容舊系統 |

## 分析與建議

- 大部分團隊選 Kafka 是對的，但不是因為 Kafka 最好 — 是因為 Kafka 生態最豐富。Kafka Connect、KSQL、Schema Registry 讓它不只是一個 queue，而是一個 data streaming 平台。
- RabbitMQ 被低估了。如果你的 QPS 在十萬級、需要複雜路由（topic exchange、header exchange），RabbitMQ 的開發體驗比 Kafka 好太多。不要迷信吞吐量數字。
- Pulsar 的存算分離架構是未來的方向，但現在還在早期。維運複雜度（ZooKeeper + BookKeeper + Broker 三層）讓小團隊吃不消。除非你需要多租戶和跨地域複製，否則先用 Kafka。
- 不要因為消息隊列很厲害就什麼都用消息隊列。一個簡單的 API call 變成異步消息，就意味著你放棄了同步的簡單性，換來最終一致性、重複消費、順序錯亂等所有分散式問題。

## 總結

**消息隊列解決了生產者和消費者的速度差，但引入了新的問題：消息丟了怎麼辦、重複了怎麼辦、順序亂了怎麼辦。選型前先想清楚你能接受哪一類失敗。**
