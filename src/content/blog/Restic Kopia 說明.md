Restic 和 Kopia 是兩個 Go 寫的開源備份工具，核心方案相似：客戶端加密 + 增量備份 + 快照管理。真正的選擇點在於：Restic 社群更大、文件更完整、存儲後端支援更廣；Kopia 多了 GUI 和壓縮功能。

> **Restic = 穩定老兵、社群最大；Kopia = 有 GUI、有壓縮、正在追趕。**

## 它解決什麼問題

- 檔案備份的三大需求：加密、去重、版本控制（快照）
- 支援多種存儲目標：本地、SFTP、S3、GCS、Azure Blob 等
- 增量備份只傳輸變更的區塊，節省空間和時間

## 功能對比

| 特性 | Restic | Kopia |
|------|--------|-------|
| 加密（端到端） | ✅ | ✅ |
| 重複數據刪除 | ✅ | ✅ |
| 壓縮 | ❌ | ✅ |
| 圖形用戶界面 | ❌ | ✅ |
| 增量備份 | ✅ | ✅ |
| 快照瀏覽/恢復 | ✅ | ✅ |
| 掛載備份為檔案系統 | ✅ | ✅ |
| 社區活躍度 | 高 | 中等 |
| 文件完整性 | 優秀 | 良好 |

## 增量備份機制（兩者相同）

```
檔案 → 分塊(Chunking) → 計算 Hash → 檢查是否已存在
                                      │
                          已存在 → 引用舊 chunk（不重複存）
                          不存在 → 上傳新 chunk
```

關鍵理解：邏輯上是全量備份（每個 snapshot 都可以獨立恢復完整檔案系統），物理上是增量存儲（只有新的或變更的 chunk 才會實際寫入）。

舉例：一個 100MB 檔案改了 1MB，第二次備份只會存那 1MB 的新 chunk，加上更新的 metadata。但從使用者角度看，你有兩個時間點的完整版本。

## 定時備份時沒變更會怎樣？

不會創建新資料，只生成一個新的 snapshot 指向上一次的資料塊。快照是輕量 metadata，不佔儲存空間。

## 分析與建議

- 個人開發者用 Restic 就夠了。CLI 不是缺點，備份這類操作本來就適合排程自動執行。
- Kopia 的 GUI 和壓縮對非技術人員很加分，尤其是想讓家人或同事也能自己管理備份的情境。
- Restic 的社群規模和文件成熟度是長期使用的重要考量 — 遇到問題時找得到解答比多一個功能重要。
- 兩者都是 Go 寫的，單一二進位檔、跨平台，部署成本幾乎為零。這在備份工具裡是很稀有的組合。
- 如果需要壓縮且不介意用 CLI，可以先評估 Restic 的 repo 大小是否在可接受範圍 — 很多場景下去重本身就夠了，壓縮的邊際效益有限。

## 總結

**Restic for production、Kopia for GUI。兩個都是 Go 寫的單二進位檔，比任何需要 daemon 的備份方案簡單一個數量級。**
