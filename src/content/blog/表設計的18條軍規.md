# 表設計的 18 條軍規

## 核心摘要

表設計決定了性能天花板。18 條軍規歸為五個面向：**命名與欄位設計**（1-4 條）、**主鍵與關係**（5-6 條）、**索引策略**（7-8 條）、**型別選擇**（9-12 條）、**可維護性規範**（13-18 條）。這不是可選的「最佳實踐」，而是線上系統不出事的底線。

## 一句話理解

**上線後改 Schema 的成本是設計時的 100 倍——寫 CREATE TABLE 的那一刻就決定了未來三年的維運痛苦程度。**

## 命名與欄位設計（1-4）

| 規則 | 要點 |
|------|------|
| 命名規範 | 小寫 + 底線分隔 + 見名知意；統一常用欄位名（create_time, update_time） |
| 欄位數量 | 單表不超過 20 欄，超過考慮垂直拆分 |
| NOT NULL | 盡量設 NOT NULL + 預設值，NULL 值會導致索引失效和額外儲存開銷 |
| 註釋 | 表和欄位都要加 COMMENT，定期更新 |

## 主鍵與關係（5-6）

| 規則 | 要點 |
|------|------|
| 主鍵設計 | 必須有主鍵；用 bigint + 無業務耦合的值（雪花算法）；不要用 UUID 字串做 PK（寫入性能災難） |
| 外鍵 | 強調一致性但高效能系統不建議使用物理外鍵，透過應用層保證 |

## 索引策略（7-8）

| 規則 | 要點 |
|------|------|
| 索引數量 | 單表不超過 5 個，包含主鍵 |
| 唯一索引 | 業務唯一欄位必須建唯一索引，注意 NULL 值的行為差異 |

## 型別選擇（9-12）

| 規則 | 要點 | MySQL vs PostgreSQL |
|------|------|---------------------|
| 欄位型別 | 固定長度用 char，可變用 varchar；數字優先用 tinyint/bit | PG 的 varchar 沒有效能差異，可不設長度 |
| 金額欄位 | 只用 decimal，禁止 float/double | PG 的 money 型不建議用，仍用 numeric |
| 時間欄位 | 用 datetime，不用 varchar 存時間 | PG 用 timestamp with time zone |
| JSON 欄位 | 結構化彈性需求可用 JSON 型 | PG 的 jsonb 比 MySQL 的 JSON 更強大，支援索引 |

## MySQL vs PostgreSQL 關鍵差異

- **字元集**：MySQL 用 utf8mb4；PG 只用 unicode，設 collation
- **唯一索引 + NULL**：MySQL 多行 NULL 不衝突；PG 多行 NULL 可重複，需小心
- **大欄位**：PG 的 text/varchar 無長度限制，大文件仍建議外部儲存
- **儲存引擎**：MySQL 用 InnoDB；PG 只有一種（heap），不用選

## 我的判斷

- **命名規範是團隊效率和程式碼可讀性的第一道防線**。一張有 20 個 `field1` `field2` 的表，交接時就是技術債。
- **主鍵用 UUID 字串是最常見的性能自殺**。UUID 的無序性會導致 B+ 樹的頁分裂遠多於自增 ID。非要用 UUID 至少用 ULID 或 UUID v7（時間有序）。
- **物理外鍵在 OLTP 系統中建議拿掉**。它不是沒有價值（保證一致性），但在高併發寫入和分庫分表場景下，物理外鍵會成為擴展瓶頸。
- 這些軍規大部分直接適用於 PostgreSQL，但 jsonb、唯一索引 NULL 行為、collation 這些細節需要單獨確認。

## 最後記住這句

**上線後的表結構變更是帶血的——你今天省下的 5 分鐘設計時間，明天會在凌晨 3 點的 on-call 裡加倍要回來。**
