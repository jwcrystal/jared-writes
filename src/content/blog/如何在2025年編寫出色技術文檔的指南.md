# 如何在 2025 年編寫出色技術文檔

## 核心摘要

這篇文章的核心論點是：大部分人寫不好文檔，不是因為不會寫，而是因為**寫給自己看，不是寫給讀者看**。五個最常見的錯誤——過多行話、步驟不完整、關鍵資訊埋藏、不更新、缺乏範例——本質上都是「假設讀者已經知道我所知道的一切」。解決方案是一個結構化框架（介紹 → 快速入門 → 核心概念 → 詳細指南 → FAQ → Changelog），加上持續更新的機制。

## 一句話理解

**讀者不知道你知道的事。文檔的目的是縮小這個差距，不是展示你的技術深度。**

## 常見錯誤

| 錯誤 | 表現 | 根源 |
|------|------|------|
| 行話過多 | 「多執行緒處理」而不是「同時處理多個任務」 | 為自己寫，不為讀者寫 |
| 步驟不完整 | 「安裝依賴項」不給具體指令 | 假設讀者知道怎麼裝 |
| 資訊埋藏 | 重要配置藏在長段落中 | 沒有使用 callout / 警告框 |
| 不更新 | 文檔版本號和實際不符 | 把文檔當一次性任務 |
| 缺乏範例 | 「如何使用 API」但沒有實際請求範例 | 用抽象描述取代實操 |

## 文檔結構框架

```
├── 項目介紹（這是什麼，解決了什麼問題）
├── 快速入門（5 分鐘內讓用戶跑起來）
├── 核心概念（關鍵術語和設計原則）
├── 詳細指南（按場景的步驟說明）
├── FAQ（常見坑和解決方案）
└── Changelog（版本變更記錄）
```

## 可讀性清單

**基本要素：**
- 清楚的標題和介紹
- 為什麼需要這個功能（不只是「是什麼」）
- 一步步的說明（不是一段落）
- 可以複製貼上的命令 / 程式碼
- 實際的輸入輸出範例

**增強體驗：**
- 警告和提示是否突出顯示
- 技術術語是否解釋清楚
- 複雜概念是否有圖表輔助
- 內容是否用標題層級清楚組織

## 四個心態轉變

| 誤解 | 正確態度 |
|------|----------|
| 「先寫 code，再補文檔」 | 開發過程中同步寫文檔，不然細節會忘 |
| 「其他工程師看懂就好」 | 為受眾寫，不是為自己寫 |
| 「越詳細越好」 | 簡潔優先，讀者會因過多資訊放棄 |
| 「文檔是一次性任務」 | 文檔有生命週期，需要持續維護 |

## 2025 年的新趨勢

- **AI 輔助**：用 AI 生成初稿和自動更新（但需要人工審核準確性）
- **互動式文檔**：嵌入可執行的 code playground 和 API 測試工具
- **多語言支援**：不只用英文，考慮目標受眾的語言

## 我的判斷

- 這篇文章最強的觀點是 **「文檔不是一次性任務」** 和 **「為讀者寫，不為自己寫」**。大部分技術文件失敗的原因就在這兩條。
- 結構化框架本質上跟這套 personal-writing-style 的設計理念一致——核心摘要先出來、結構讓掃讀者能快速定位、細節按層級展開。
- 「先寫 code 再補文檔」是效率殺手。寫 code 時你的 context 是滿的，隔一週再回來寫文檔，至少丟掉 50% 的細節。同步寫的品質和效率都更好。
- AI 輔助寫文檔現在已經很好用了（Cursor、Claude 可以直接從 codebase 生成 API doc），但需要注意兩點：AI 不知道產品的「為什麼」，以及 AI 的 docs 很容易過時（生成後沒人更新）。

## 最後記住這句

**好文檔的標準不是「寫了多少字」，而是「讀者能不能在 5 分鐘內跑起來，並在遇到問題時 30 秒內找到答案」。**
